Quick start

Fast setup

1

Build the payload

Build the payload for the target platform, or the host build for local testing.

make host
make payload-ps5 PS5_PAYLOAD_SDK=/path/to/ps5-payload-sdk
2

Start MemDBG

On host builds, you can pass the port, bind address, and log destination from the command line.

./build/memdbg-host --bind=0.0.0.0 --debug-port=9020 --udp-port=9023
3

Open the frontend

Build and launch the native interface, then connect to the console or host IP address.

make frontend
./build/frontend/memdbg_frontend

Setup

Payload preparation

MemDBG exposes a TCP server for commands and a UDP channel for logs. The payload must be running before you use processes, maps, memory reads, scanners, or trainers.

Useful arguments

ArgumentPurpose
--bind=0.0.0.0Address the payload listens on.
--debug-port=9020TCP port used by the frontend.
--udp-host=255.255.255.255UDP host or broadcast address for logs.
--udp-port=9023UDP port for live console logs.
--data-root=/data/memdbgPayload data and log directory.
--no-udp-logDisable UDP log delivery.

Console permissions

On supported platforms, the privilege module tries to elevate the payload at startup and temporarily elevate the target process during read/write/batch operations. Look for a log line similar to privilege: payload escaped sandbox. If that line is missing or the operation fails, many actions can return payload status -3 or payload status -8.

Frontend

Connection and settings

Connection

Enter the console or host IP address, confirm the TCP port, and press Connect. After the HELLO response, the sidebar shows the connection state, IP, port, and UDP listening status.

Saved IP

In Settings, you can save the IP address you use most often. This is useful when the console has a static address or a DHCP reservation on your router.

Console log

The Logs screen shows UDP messages from the payload. If messages do not arrive, verify that the frontend is listening on the same UDP port configured in the payload, and that the firewall or network is not blocking broadcast traffic.

Telemetry

The Telemetry screen summarizes uptime, active connections, total reads/writes, and scanner cache performance. Use it to understand whether a trainer is writing too often or a scan is producing many read errors.

Workflow

Processes, maps, and memory

  1. Connect to the payload and open the Processes section.
  2. Refresh the process list and select the game or application process.
  3. Load the memory maps for the selected process.
  4. Prefer maps with read permission for scans and memory reads.
  5. Use Dump selected map to save a region to a chosen location.

Rule of thumb

Before reading or scanning, you need a valid process and, for range scans, a range that matches the process maps. Bad ranges, terminated processes, or unreadable pages can produce I/O errors even when the TCP connection is still alive.

Memory

Read, write, and batch

Read Memory

Reads bytes from an absolute address in the selected process. Use small lengths while validating an address, then increase the length only when the range is stable.

Write Memory

Writes bytes into the target process. Before writing, always capture the original value: you will need it to restore the state or build a safe trainer toggle.

Batch

Batch operations read or write multiple addresses in a single request. They are ideal for trainers with several active cheats and reduce TCP traffic compared with many separate requests.

Scanner

Finding useful addresses

Exact scan

Searches for an exact value in a range or across the whole process. Choose the correct type: u8, u16, u32, u64, f32, or f64. If you are unsure, start with u32 for integer counters or f32 for decimal values.

Process scan

Scans the readable maps of the selected process. It is useful when you do not know the correct region, but it can be slower and can produce more errors on protected or unstable maps.

AOB scan

Searches for a byte signature. Use wildcards for variable bytes and build patterns that are long enough to avoid too many matches. This is a strong base for trainers that need to survive address changes.

Pointer scan

Use pointer chains when an address changes after a restart or level change. Always verify the chain by restarting the game or reloading the save.

Recommended method: scan a known value, change that value in game, narrow the results with a new scan, then verify candidates by reading and writing controlled values.

Trainer

Building a trainer

Example: infinite health

  1. Select the game process.
  2. With health at 100, run an exact scan for u32 = 100.
  3. Take damage, for example health at 83, and scan for u32 = 83.
  4. Repeat until only a few results remain.
  5. Read the candidate and verify that it changes with the in-game value.
  6. Add the cheat in the Trainer section.
  7. Capture the original OFF value.
  8. Set the ON value, for example 999, and enable lock if needed.

Cheat fields

NameReadable label, for example Infinite Health.
AddressAbsolute address or a result resolved from AOB/pointer data.
TypeValue type: byte, integer, float, or raw bytes.
ON valueValue written when the cheat is enabled.
OFF valueOriginal value restored when the cheat is disabled.
LockPeriodically rewrites the ON value.

Stable trainers

Absolute addresses can change on every launch because of ASLR, updates, or version differences. For trainers you want to share, prefer AOB signatures or pointer chains verified across multiple sessions.

OFF value

Capture OFF before enabling the cheat. If the original value is dynamic, disabling the cheat may not be enough: in that case, use lock carefully and test restoration on an unimportant save.

Dump

Saving a memory map

After selecting a process and loading its maps, choose a readable map and use Dump selected map. Pick an explicit path to save the dump in the location you want. Dumps are useful for offline analysis, version comparison, and AOB signature creation.

Watch the size

Some maps can be very large. If the dump fails, try a smaller region or verify the permissions on the selected map.

Troubleshooting

Common errors

ErrorMeaningWhat to check
payload status -3 I/O error during the requested operation. Selected process, valid range, readable map, payload privileges, and whether the process is still alive.
payload status -8 Permission denied. Privilege module, active jailbreak, payload logs, and target process permissions.
not connected The frontend does not have an active TCP session. Connect to the console first and verify IP/port in Settings.
Empty process list The payload cannot enumerate processes or is not connected. Ping, payload logs, supported platform, and process privileges.
Scan has no results The value, type, or range does not match. Try another type, narrow to the correct maps, use AOB, or use unknown scan.
No UDP logs The frontend and payload are not communicating on the UDP channel. Port 9023, firewall, broadcast, and same network segment.

Best practice

Safe and repeatable usage

  • Use disposable saves when testing writes or trainers.
  • Record the game version, title id, and payload build for every trainer.
  • Before using lock/freeze, verify that the game is not also writing the value every frame.
  • Prefer AOB and pointer chains for trainers intended to work across sessions.
  • Check Telemetry: many read errors usually mean the range is too wide or maps are unreadable.
  • Keep TCP and UDP on different ports to avoid conflicts with other payloads or tools.
